# ArmorPilot License Activation Guide

This guide is for customers and console administrators who need to request,
install, verify, renew, or recover an offline license.

It does not contain the vendor private-key procedure. License issuers should use
[`LICENSE_ISSUER_GUIDE.md`](LICENSE_ISSUER_GUIDE.md).

For editions and commercial pricing, see
[`LICENSE_PRICING.md`](LICENSE_PRICING.md).

## 1. How Offline Activation Works

Each commercial installation has a unique Installation ID. The ID is derived
from:

- An installation key generated by the console on first use.
- A persistent installation UUID.
- The `kube-system` namespace UID.
- The Kubernetes API CA certificate fingerprint.

The console produces a signed activation request. The vendor uses that request
to issue a `ARMORPILOT1...` license key bound to this installation.

```text
Console installation
        |
        | Download Activation Request
        v
varmor-activation-request.json
        |
        | Send to license vendor
        v
Vendor-issued ARMORPILOT1 license key
        |
        | Paste into Install License
        v
Activated console
```

No activation server or internet connection is required.

## 2. Request a New License

Sign in with an account that has `license:view`.

1. Open **Users**.
2. Select the **License** tab.
3. Confirm that **Installation ID** is displayed.
4. Click **Download Activation Request**.
5. Save the downloaded `varmor-activation-request.json`.
6. Send the file to the license vendor together with:
   - Legal customer or organization name.
   - Requested edition.
   - Required node count.
   - Estimated maximum policy count.
   - Requested subscription period.
   - Purchase order or contract reference, if applicable.

Do not edit the activation-request file. It contains a signature, and any
modification causes the vendor verification to fail.

### Alternative: Copy Installation ID

The **Copy Installation ID** button is useful for quotation and support
communication. The complete activation-request file is still required to issue
a production-bound license because it proves possession of the installation
private key and includes the cluster identity details.

## 3. Activate the License

The vendor returns a single-line key beginning with:

```text
ARMORPILOT1.
```

To activate:

1. Sign in with an account that has `license:manage`.
2. Open **Users > License**.
3. Paste the complete `ARMORPILOT1...` key into **Install License**.
4. Click **Save License**.
5. Wait for **License saved and verified**.
6. Click **Refresh** if the status panel does not update immediately.

Successful activation should show:

- `Status`: `valid` or `in_grace`.
- `Bound license`: `yes`.
- The licensed customer and edition.
- License expiration date.
- Node and policy limits.
- The same Installation ID used in the activation request.

The console stores the verified license at `/app/data/license.json` by default.
Production trials are delivered as short-lived vendor-signed keys. The
installation-date built-in trial is disabled in production container builds.

## 4. Verify Activation

In the License status panel, confirm:

| Field | Expected value |
|---|---|
| Status | `valid` |
| Compliant | `yes` |
| Bound license | `yes` |
| Installation ID | Current installation ID |
| License ID | ID issued by the vendor |
| Customer | Licensed customer |
| Nodes | At or below `max_nodes` |
| Policies | At or below `max_policies` |

An administrator may also check the API:

```bash
curl -u '<username>:<password>' \
  http://<console-host>:30080/api/license
```

Important response fields:

```json
{
  "status": "valid",
  "valid": true,
  "binding_required": true,
  "payload": {
    "license_id": "LIC-CUSTOMER-001",
    "installation_id": "api_..."
  },
  "installation": {
    "available": true,
    "installation_id": "api_..."
  }
}
```

The two `installation_id` values must match.

## 5. Common Activation Errors

### `license key or JSON is invalid`

The pasted content is incomplete or is not a `ARMORPILOT1...` key.

Action:

- Paste the entire key as one line.
- Do not paste the public key.
- Do not add quotes or Markdown code fences.

### `license signature is invalid`

The key was modified, truncated, or signed by a different vendor key.

Action:

- Request the original key again.
- Confirm that the console image contains the correct vendor public key.

### `license is not bound to this installation`

The license is an old or unbound key while commercial binding is required.

Action:

1. Download a new activation request.
2. Ask the vendor to issue a bound replacement key.

### `license installation_id does not match this installation`

The key belongs to another console installation.

Action:

- Do not reuse a key from another cluster.
- Send the current installation's activation request to the vendor.
- Use the rehost procedure if this is a legitimate migration.

### `installation identity is unavailable`

The console cannot read or create its installation identity.

Check:

- `/app/data` is writable and persistent.
- The service account can read the `kube-system` namespace.
- The Kubernetes service-account CA file is mounted.
- The console logs contain no storage or Kubernetes API errors.

### `license limits exceeded`

The runtime node or policy usage exceeds the signed limits.

Action:

- Reduce usage where appropriate, or
- Request a license upgrade with higher limits.

## 6. Renew a License

For a normal renewal on the same installation:

1. Open **Users > License**.
2. Download a fresh activation request.
3. Send it to the vendor with the renewal order.
4. Receive the renewed `ARMORPILOT1...` key.
5. Paste it into **Install License** and click **Save License**.

Installing the renewed key replaces the previous license. Existing Kubernetes
policies are not removed.

Recommended timeline:

- 60 days before expiration: request a renewal quotation.
- 30 days before expiration: complete the renewal request.
- Before the grace period ends: install the renewed key.

## 7. Rehost or Move to Another Installation

A bound key cannot be reused after moving to an installation with a different
Installation ID.

For a legitimate migration:

1. Deploy the console on the new cluster.
2. Open the License tab on the new installation.
3. Download its activation request.
4. Send the new request and the old License ID to the vendor.
5. Request a rehost/replacement key.
6. Install the replacement key on the new installation.
7. Retire the old installation according to the license agreement.

Do not copy only `license.json` to the new installation. It will fail identity
verification.

## 8. Backup and Disaster Recovery

The installation identity and installed license are stored under `/app/data`:

```text
/app/data/installation-private.pem
/app/data/installation.json
/app/data/license.json
```

Back up this directory securely.

Restoring the same `/app/data` backup together with the same cluster identity is
treated as disaster recovery of the original installation. The private key must
remain confidential and must not be copied to a second simultaneously active
installation.

If `/app/data` is lost:

1. The console generates a new Installation ID.
2. The previous license no longer matches.
3. Download a new activation request.
4. Ask the vendor for a replacement license.

## 9. Expiration Behavior

- At 30 days remaining, the console displays a warning.
- After `expires_at`, the license enters its signed grace period.
- After the grace period, the license becomes invalid.
- Existing Kubernetes policies are not automatically deleted.
- With fail-closed licensing, new policy creation/import/restore/approval is blocked.

## 10. Security Notes

- Never request or receive the vendor private signing key.
- Do not share the installation private key.
- Do not edit activation requests or license keys.
- Use one bound license per active installation.
- Keep `/app/data` on persistent storage with restricted access.
- Offline licensing cannot detect two simultaneous exact clones if the complete
  cluster state and `/app/data` are duplicated.
